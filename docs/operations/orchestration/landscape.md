# Agent Orchestration Landscape — Claude Code (May 2026)

Scope: every reasonable way to coordinate AI agent work *from or around Claude Code*. Native primitives first, then third-party orbiters. Honest about experimental status and where cost data is thin.

## Platform constraints to read before the table

These bind every native pattern below. A recommendation that violates one of them is platform-invalid (or, for #6, billing-invalid) — not just suboptimal.

1. **Subagents cannot spawn subagents.** Only the top-level Claude Code session can fan out via the `Agent` tool. Any architecture shaped like "supervisor subagent dispatches worker subagents" does not run; the supervisor will improvise inline and the deviation is silent unless a hook catches it. (Source: `sub-agents` docs — subagents return results to *the caller*; nesting isn't part of the contract.)
2. **Agent-team teammates cannot spawn teams.** Same rule, restated: only the lead can manage the team. Teammates can use the `Agent` tool to spawn ordinary subagents *within their own session*, but they cannot create teammates of their own. (Source: `agent-teams` docs, Limitations: "no nested teams".)
3. **Agent teams do not auto-worktree teammates.** Two teammates editing the same file overwrite each other. File partitioning is the user's job. (Source: `agent-teams` docs, "Avoid file conflicts".)
4. **Custom-agent frontmatter `skills` and `mcpServers` are dropped when the def runs as a teammate** (they apply only when it runs as a subagent). Same role, different loaded context, by design.
5. **Channels and agent view are research preview**; agent teams are experimental (env-var gated). Treat as not-production unless you've tested for your case. **Channels on Team/Enterprise plans require admin enablement** (`channelsEnabled: true`) — for a personal claude.ai / Pro / Max account this is automatic, but for any org-managed account treat channels as unavailable until the admin opts in.
6. **Agent SDK billing changes on 2026-06-15 (27 days from today, 2026-05-19).** From that date, Agent SDK and `claude -p` usage on subscription plans draws from a *separate* monthly Agent SDK credit pool, distinct from the interactive Claude Code limit. Any "build a custom orchestrator in the SDK" recommendation needs to budget for that separate credit, not assume it pulls from the same bucket as `claude` sessions. (Source: `agent-sdk/overview` docs, top-of-page note.)

## Deeper notes per pattern

**Single-session sequential.** This is the baseline. You can already get a lot of parallelism *within* a single turn by issuing multiple independent tool calls in one assistant message — those execute in parallel. People reach past this only when (a) the work would blow the context window, (b) different parts of the work need different system prompts or tool allowlists, or (c) calendar-wallclock matters (overnight runs). If none of those apply, the orchestration overhead loses to a single session.

**Agent tool subagents.** The most under-appreciated primitive. Three built-in types (`general-purpose`, `Explore`, `Plan`) cover the common cases. Explore and Plan deliberately skip `CLAUDE.md` and git status to keep their context small, which is also why a forked skill using `agent: Explore` is so cheap. Hard limit: subagents can only report back to the agent that spawned them. They can never message each other. If two subagents in the same session need to coordinate, the parent has to be the courier.

**Custom subagents.** A definition is a markdown file with frontmatter at `.claude/agents/<name>.md` (project) or `~/.claude/agents/<name>.md` (user). Frontmatter accepts `description`, `tools`, `model`, `skills`, `mcpServers`, `isolation: worktree`, and a few others. The body becomes the system prompt. The double-life consideration: when an agent def is referenced as an agent-team teammate, its `skills` and `mcpServers` are **not** applied — teammates inherit those from the project/user settings instead. If your role's identity depends on a custom skill being preloaded, it'll behave differently in a team than as a subagent. *Recurring-cost honesty:* each definition's `description` sits in the subagent listing on every session in scope — adding ten specialists for cases that fire once a quarter taxes every session, not just the ones that use them.

**Agent teams.** The most interesting native pattern, and the most expensive. Docs say "significantly more tokens" but don't publish numbers — the rough mental model is N teammates ≈ N+1× a single session's token cost, plus messaging overhead. *That "N+1×" is a back-of-envelope ordering hint, not a budget number — do not plan against it as if it were measured.* Architecture is straightforward: shared task list with file-locking for race-free claims, a mailbox for `SendMessage`, and a lead that's fixed for the team's lifetime. Limitations to take seriously: in-process teammates don't survive `/resume`; teammates don't get worktrees automatically (so file conflicts are a real risk and you have to partition work by file); no nested teams; no per-teammate permission modes at spawn; task-completion lag can block dependent tasks. The Anthropic-recommended sweet spot is 3–5 teammates and 5–6 tasks per teammate.

**Hooks (enforcement).** The hooks reference enumerates ~28 events covering session, per-turn, tool, subagent, task, config, compaction, worktree, MCP elicitation, and notifications. The orchestration-relevant ones are `TaskCreated`, `TaskCompleted`, `TeammateIdle`, `SubagentStop`, `Stop`, `PostToolBatch`, `UserPromptSubmit`, `SessionStart`. All of those can `decision: "block"` or exit code 2 to block-with-feedback. `PreToolUse` is the most powerful single hook — it can deny, ask, defer, *modify the tool input*, *create permission rules on the fly*, and *inject additional context*. That alone is enough to build substantial enforcement logic without writing any orchestrator code.

**Hooks (orchestration glue).** You can build a surprising amount with hooks alone: `TaskCompleted` → run tests → block if red; `TeammateIdle` → check shared store → push more work or release; `SessionStart` → fetch CI/Linear state → inject as additional context. The honest limit is debuggability. There's no DAG visualisation, no central scheduler, no retry policy — it's pure event-driven scripts, and once you have more than ~3–4 cooperating hooks the state becomes implicit. At that point, the Agent SDK is cheaper to maintain than the hook web.

**Custom skills.** Two distinct cost profiles. Reference-style skills (description + body that becomes part of context) add a sticky cost — once invoked, the body lives in the conversation. Forked skills (`context: fork`, `agent: Explore`) cost like a subagent: ephemeral and disposable. The 25k-token re-attachment budget post-compaction matters for long sessions: if you've invoked 5 skills and they total more than 25k tokens, the older invocations get dropped after compaction, ranked by recency. Use `disable-model-invocation: true` for skills with side effects (`/deploy`, `/commit`) — otherwise Claude can decide to call them, which you almost never want. *Recurring-cost honesty:* even unused skills pay the description-budget tax on every session in scope (budget scales at 1% of the model's context window per the skills troubleshooting docs). On overflow, **all skill *names* stay listed, but descriptions for the skills you invoke least are dropped first** — keywords disappear, the model stops auto-matching those skills, and you debug a routing bug that's actually a budget bug. `/doctor` shows whether the budget is overflowing and which skills are affected. The listing already trims at the source (1,536-char cap per skill's combined `description` + `when_to_use`, configurable via `maxSkillDescriptionChars`; budget itself raisable via `skillListingBudgetFraction` or `SLASH_COMMAND_TOOL_CHAR_BUDGET`).

**Slash commands as orchestration triggers.** They're triggers, not runtimes. The interesting bundled ones for orchestration: `/batch` (auto-fans-out a single instruction into N worktree-isolated subagents, each opens a PR); `/agents` (manager for subagent definitions and running subagents); `/tasks` (lists background work in current session); `/background` (detaches the whole session to keep running); `/loop` and `/schedule` (skill-based scheduling — `/loop 5m /foo` runs `/foo` every 5 min).

**MCP servers as orchestration primitives.** This is the under-documented escape hatch. If you need a primitive the runtime doesn't give you (durable job queue, distributed lock, cross-session shared state beyond the team mailbox, external scheduler with retries), the recommended path is to expose it as MCP tools. Channels are a special case: MCP servers that *push* events into a live session rather than waiting for tool calls. Channels are research-preview, plugin-allowlist-gated, and don't work on Bedrock/Vertex/Foundry. But they're the only way for an external system to *initiate* work in a running Claude Code session without you typing. *Recurring-cost honesty:* every tool an MCP server exposes occupies tool-listing slots on every session that loads the server, and a chatty server's descriptions can crowd out skill descriptions and subagent descriptions. The right move is per-project `.mcp.json` scoping, not a fat user-level config that loads everywhere.

**Claude Agent SDK.** Everything Claude Code does — agent loop, sessions, hooks, subagents, MCP, skills, slash commands, plugins, streaming, structured outputs, file checkpointing, custom system prompts — is exposed as Python or TypeScript primitives. You own the harness. The migration guide (the SDK was renamed from "Claude Code SDK" to "Claude Agent SDK") and the v2 preview-removed note suggest the API surface is still moving. Critical billing change to flag: **from June 15, 2026, subscription-plan Agent SDK and `claude -p` usage draws from a separate monthly Agent SDK credit pool**, not the same interactive limits as Claude Code itself. This makes "is this a CLI workflow or an SDK workflow" a real cost question, not just an interface question.

**Git worktrees + manual parallel sessions.** The simplest "multi-agent" pattern that exists: `claude --worktree foo` in one terminal, `claude --worktree bar` in another. Each gets `.claude/worktrees/<name>/`, a fresh branch from `origin/HEAD` (configurable to local HEAD via `worktree.baseRef: "head"`), and `.worktreeinclude` lets you copy gitignored files (`.env`, etc.) into each. Cleanup is automatic only when there are no changes — otherwise you're prompted. Worktrees solve file-conflict isolation; they do *nothing* for inter-agent communication. Combine with agent view (for dashboarding) or agent teams (for messaging).

**Agent view (`claude agents`).** Research preview. The fleet-management UX: one screen showing every background session, its state, which ones want input. Auto-worktrees per session. The mental model is "delegated work I check back on", which sits between subagents (in-session) and agent teams (coordinated). Sessions in agent view report only to you, not to each other.

**Claude Squad and the local-fleet family.** Pure-local TUIs that wrap `tmux` + worktrees + multiple terminal agents. Visibility-first: you see N panes, switch with keystrokes. Claude Squad specifically is multi-agent-runtime (works with Claude Code, Codex, OpenCode, Amp). Trade-offs vs. agent view: simpler to install, runtime-agnostic, but no crash recovery and you're on your own for state. Use when you want fleet visibility without writing tmux config; skip when you'd benefit from agent teams' built-in mailbox/task-list.

**opencode.** Not Claude Code; a different open-source agent runtime that *can* use Claude as one of many providers. Worth knowing about if (a) you want provider freedom, (b) you want background subagents and a research-focused "Scout" agent in the same tool, or (c) you'd rather pay your own provider directly than route through claude.ai. It does not read `.claude/` config and doesn't share the skill/hook ecosystem, so it's a *replacement* for Claude Code, not an *addition*.

**Aider.** Single-agent, but the **architect mode** is the genuinely useful orchestration idea: a "thinking" model writes the plan and a cheaper "editing" model executes it, claimed 30–50% cost saving vs. running the strong model end-to-end. You could approximate this in Claude Code with a Plan-mode subagent followed by a Sonnet/Haiku editor subagent, but Aider has it as a first-class workflow. No subagents, no hooks, no parallel agents — its sweet spot is single-developer flow, not orchestration.

**Roo Code.** VS Code-bound. Orchestrator mode ("Boomerang Tasks") is genuinely a multi-agent pattern, where one mode decomposes and delegates to specialised modes, each of which can use a different model/provider. Not a Claude Code competitor in CLI/CI workflows, but the per-mode-model-choice idea is the orchestration angle worth borrowing — it's effectively "agent teams where each teammate's model is part of the team design", which Claude Code agent teams only support globally (via "Default teammate model" in `/config`).

**Other tmux orchestrators.** Conductor, amux, Vibe Kanban, dmux, Antfarm/OpenClaw, Gastown. Variations on "many local agents + worktrees + dashboard". Differentiators are persistence (amux for overnight runs), multi-device access, automatic crash recovery, and how opinionated the workflow is. Most are 2025–2026 projects with rapidly shifting feature sets — if you commit, plan to re-evaluate every 3–6 months.

**Managed Agents.** Sibling to the Agent SDK, with the inverse trade-off: Anthropic runs the loop and the sandbox; your code sends events over HTTP. Doesn't work on your local files (the sandbox is a managed checkout), so it's a *production* deployment target, not a local dev tool. Prototype with the SDK, move to Managed Agents when you want someone else operating the sandbox and session log.

## Honest gaps in this catalogue

- **No published per-pattern token numbers from Anthropic.** "Significantly more tokens" for agent teams is the strongest official statement; the linked `/en/costs#agent-team-token-costs` anchor was not fetched in this pass. Treat the cost column as relative ordering, not measurement. **The "N+1×" mental model for agent teams is a back-of-envelope ordering hint, not a budget number — downstream docs (e.g., pattern-map) should not plan against it as if it were measured.**
- **Channels and agent view are research preview.** Both could ship with breaking changes; agent teams are explicitly experimental and the docs list nine current limitations.
- `**/batch` is described in docs in a partly future-tense way ("a planned split…").** I treated it as currently available because it's listed among bundled skills, but the wording suggests features may still be landing.
- **Third-party tool descriptions reflect mid-2026 search snapshots, not first-hand testing.** The orchestrator-ecosystem area (Claude Squad, amux, Conductor, Antfarm, etc.) is moving fast and any of these could be deprecated or merged by the time the recommendation is acted on.
- `**SubagentStart` and `SubagentStop` exist in the hook reference even though docs say subagents "only report back".** This means you *can* observe subagent lifecycles from outside the parent agent — useful for audit and metrics, but not for inter-agent communication.
