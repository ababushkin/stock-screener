<title>/stock:model — playbook loader for watchlist tickers (depth-over-breadth foundation)</title>
<description>
## Problem

`/stock:model` currently applies the same generic ESTABLISHED / EMERGING logic to every ticker. That's the breadth-instinct showing through. For the core watchlist (GOOG, META, AMZN, NVDA, ASML, NFLX — see `WATCHLIST.md`), we need ticker-specific assumptions baked in:

* Business architecture and segment splits (META FoA + RL, GOOG Search + Cloud + YouTube, AMZN AWS + Ads + Retail, etc.)
* Capex-cycle position and implied normalised margins
* Catalyst-aware scenario narratives (not generic "revenue beat / miss")
* Sell-side disagreement axes as scenario axes
* Per-ticker WACC, growth ceiling, terminal margin overrides
* Historical reset-and-recover priors as confidence inputs

Without a loader, this knowledge lives in conversation context (lost across sessions) or in user heads (lost across humans). We need it in versioned artifacts.

## Why now

User-stated operating principle (`CLAUDE.md` → "depth over breadth"): *"world's best decision-making engine for a curated tech watchlist, not a generalised valuation tool for thousands of tickers."* The playbook loader is the infrastructure that makes this principle executable.

Blocked by <issue id="89e1f4f4-8b04-4c5b-9931-7f4f837de45f">ABA-110</issue> (SBC strip) and <issue id="e78e36cd-49f1-441f-96a9-317acc594230">ABA-111</issue> (growth-rate ceiling) — playbook overrides on a broken base produce noise on top of bias.

## Scope

**Playbook file format:** `playbooks/TICKER.md` — markdown with structured YAML frontmatter for machine-readable overrides + prose for narrative content. Structure documented in `WATCHLIST.md` → "Playbook structure (target)".

**Loader integration in** `/stock:model`**:**

1. After GATE passes, before GATHER, check `playbooks/TICKER.md` (uppercase ticker).
2. If absent: emit `No playbook for TICKER — using generic ESTABLISHED/EMERGING logic.` and proceed unchanged.
3. If present: load the YAML frontmatter. Surface to the user: `Playbook loaded: playbooks/TICKER.md (last updated YYYY-MM-DD by COMMIT_AUTHOR). Overrides applied: [list].` Continue to GATHER.

**Override hierarchy (playbook wins, generic falls back):**

* `base_wacc` — overrides the MIP-asked default (suppress the question when playbook supplies a value); user can still override at runtime via `--wacc=X.X`
* `growth_ceiling` — overrides the 18% generic fallback in <issue id="e78e36cd-49f1-441f-96a9-317acc594230">ABA-111</issue>
* `terminal_margin` — overrides the SBC-stripped TTM margin as the Y5 anchor
* `bear_narrative` / `base_narrative` / `bull_narrative` — replace generic per-scenario one-liners
* `scenario_axes` — optional, advanced: per-scenario override of growth/margin/WACC if the catalyst structure demands it (e.g. "TikTok divested" is a discrete outcome, not a smooth scenario)
* `failure_modes` — surfaced as a separate output section: `Monitor: [list]`
* `confidence_anchor` — playbook-set max confidence; e.g. NVDA caps at MEDIUM until customer concentration risk is resolved

**JSON contract addition:**

```json
"playbook": {
  "loaded": true,
  "path": "playbooks/META.md",
  "last_updated": "2026-05-17",
  "overrides_applied": ["base_wacc", "growth_ceiling", "base_narrative"],
  "failure_modes": ["..."]
}
```

When no playbook: `"playbook": {"loaded": false}`.

## Acceptance criteria

1. **Loader fires for watchlist tickers:** `/stock:model META` after a sample `playbooks/META.md` is committed emits the "Playbook loaded" line and applies overrides. Running on a non-watchlist ticker (e.g. TSLA) emits "No playbook" and runs generic logic unchanged.
2. **Override hierarchy verified:** ticker-specific WACC suppresses the MIP question; growth ceiling override changes the cap source from "fallback_ceiling" to "playbook"; narrative override appears verbatim in the OUTPUT block.
3. **Audit trail intact:** JSON `stages.model.playbook.overrides_applied` lists every field that came from playbook (not generic). Confidence cap from playbook is the binding cap when stricter than the runtime calculation.
4. **No-playbook path unchanged:** existing acceptance criteria for `/stock:model` (<issue id="50d6773f-2236-411c-9e79-2fe92ca6619f">ABA-31</issue>, <issue id="e39fa8ff-7603-4416-befd-42169bf87cdc">ABA-34</issue>, <issue id="2f80cc33-81ba-48c3-8657-2f5843994e72">ABA-93</issue>) still pass on tickers without a playbook.
5. **Playbook validation:** loader rejects malformed YAML with a clear error; warns on unknown frontmatter fields (forward-compatible) but does not fail.
6. **Documentation:** `WATCHLIST.md` "Playbook structure" section becomes a working spec with a minimal example playbook (probably `playbooks/META.md` as the seed).

## Out of scope (separate tickets per watchlist name)

* Writing the six actual playbooks — each is its own research + write-up ticket (\~1-2 days). This ticket builds only the loader infrastructure plus one seed playbook (likely META, since we just ran it) to prove the system.
* Segment-level DCF buildup (the "FoA + RL" framing) — depends on playbook system but is a separate methodology axis; one ticket after this one lands.
* Backtesting harness — separate; depends on playbooks existing for the full watchlist.

## Sequencing rationale

Land **after** <issue id="89e1f4f4-8b04-4c5b-9931-7f4f837de45f">ABA-110</issue> and <issue id="e78e36cd-49f1-441f-96a9-317acc594230">ABA-111</issue>. Then this ticket. Then six playbook tickets in parallel (each independently writable once the loader exists). Segment-level DCF as a follow-up methodology axis. Backtesting last, once the playbooks have settled.
